# Effect Billing Foundation Implementation Prompt

このリポジトリに追加実装するときは、以下の設計意図を最優先で守ること。

## 目的

- このプロジェクトは「外はオーダーメイド、内は共通化」の業務アプリ基盤である。
- 最優先は「最初の 3 案件を高速に回せる骨格」を育てること。
- 理想的なフレームワークを作るのではなく、今すぐ売れる最小基盤を着実に伸ばすこと。

## 判断基準

- 顧客ごとの差分は残してよい。ただし共通 workflow に持ち込まない。
- 共通化は「2社目・3社目でもほぼ確実に再利用するもの」から行う。
- 迷ったら、過剰抽象化よりも差し替えやすい単純実装を選ぶ。
- 新しい概念や抽象を増やす前に、既存の `domain` `schema` `services` `workflows` `layers` `adapters` に収まらないか確認する。

## ディレクトリ責務

- `src/domain`: 業務の中心型、状態遷移、純粋関数を置く。ここでは外部 API や永続化を知らないこと。
- `src/schema`: 外部入力を decode / validate / normalize した後の内部モデルを定義する。
- `src/adapters/customers`: 顧客固有の入力形式、マッピング、個別ルールの吸収口。CSV 列名や顧客依存の項目名はここで止める。
- `src/services`: Repo や外部サービスの抽象を置く。SDK や DB 実装の詳細を漏らさない。
- `src/layers`: `services` の live / test 実装を束ねる。外部依存の差し替え点にする。
- `src/workflows`: Effect 合成で業務フローを書く。共通 workflow は正規化済みモデルだけを扱う。
- `src/app`: 実行入口。デモや将来の API 起動コードを置く。

## 守るべき実装ルール

- 外部入力は必ず `Schema` を通して decode / validate / normalize する。
- 顧客差分は `adapters/customers` で吸収し、`workflows` に生の入力形式を渡さない。
- 状態遷移の整合性は `domain` で守る。workflow 側で場当たり的に `status` を直接書き換えない。
- 外部依存は `services` と `layers` に閉じ込める。workflow や domain から SDK を直接呼ばない。
- 失敗は単なる `throw` で終わらせず、既存の型付きエラーに寄せる。必要なら新しいエラー型を追加する。
- 監査ログが必要な業務イベントは `WorkflowEvent` と `WorkflowLogger` を通す。
- リトライはメール、決済、AI、外部 API、DB などの I/O 境界で検討する。純粋関数や業務ルールには入れない。
- AI は独立した魔法ではなく workflow の 1 ステップとして扱う。`AiGateway` の外にプロンプト生成やモデル依存を漏らさない。

## Effect の使い方

- 長い業務フローは `Effect.gen` を優先する。
- 小さな変換や decode パイプラインは `pipe` を使う。
- 純粋な値変換まで無理に Effect 化しない。
- test しやすさのために、時間・外部通信・永続化は Layer 経由にする。

## 今はやらないこと

- 早すぎる DSL 化
- 汎用プラグイン基盤の作り込み
- 案件数が少ない段階での過度な設定駆動化
- 将来使うか不明な抽象の先回り実装
- UI の先行肥大化
- microservices 化

## 追加実装時のチェックリスト

- この変更は共通部品か、顧客差分かを明確に説明できるか。
- 顧客差分が共通 workflow に漏れていないか。
- 新しい外部依存は `services` / `layers` の後ろに隠れているか。
- 入力境界で `Schema` による検証と正規化がされているか。
- 主要な状態変化と失敗が監査ログに残るか。
- テストで差し替え可能な構造になっているか。
- その抽象は本当に今必要か。単純な実装で十分ではないか。

## 変更の進め方

1. まず最小のユースケースを通す。
2. 次に共通化ポイントが本当に見えた部分だけ抽象化する。
3. 顧客差分は adapter 側へ押し戻す。
4. workflow は読みやすさを優先し、業務の流れが追える形で保つ。

## 期待するアウトプット

- 小さく動く実装
- 既存骨格に沿った責務分離
- テスト可能な構造
- 次の 2 社目導入で再利用しやすい差し替え点

迷ったら「最初の案件で本当に使えるか」「2社目でどこを差し替えるだけで済むか」を基準に判断すること。
